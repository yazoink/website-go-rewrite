The first entry in this blog was about my moderately scuffed content management setup. You can read the full post if you want all the context but tldr; I made it so all of the posts (including their contents) are read directly from a JSON file.

This system works fine in theory, but in practice it's extremely annoying to write HTML inside a JSON file.


My solution to this problem was actually very simple and I don't know why I didn't think of this initially. I also simplified a few other things, as you'll see.


The original JSON was formatted like this:

[
 {
  "title": "Example Blog Entry",
  "entry": 1,
  "subheading": "This is a blog post",
  "date": "22 September 2024",
  "categories": ["Personal", "Technology"],
  "content": [
   "<p>This is a paragraph.</p>",
   "<p>This is another paragraph.</p>"
  ]
 }
]

And the new file is formatted like this:

[
  {
    "title": "Example Blog Entry",
    "subheading": "This is a blog post",
    "date": "22 September 2024",
    "categories": ["Personal" "Technology"]
  }
]

Obviously a lot simpler!

I moved the blog posts to individual HTML files in their own directory and made the code for the blog generate a post ID of sorts (instead of specifying one). It does this by taking the title variable and replacing all the spaces with hyphens and converting all the capital letters to lower case.

All of the files in the blog/entries directory are named as <ID>.html and that's how the script searches for them.


The new directory structure looks like this: 
blog/
|
|_ entries/
|  |_ example-blog-post.html
|  |_ another-blog-post.html
|
|_ images/
|  |_ image_from_a_blog_post.jpg
|  |_ another_image_from_a_blog_post.jpg
|
|_ blog.json

The IDs are also used in the page URLs, making the old entry variable unnecessary.


This has made the process of making posts substantially easier, so hopefully I'll use this thing more now.
